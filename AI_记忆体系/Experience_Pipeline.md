# Experience_Pipeline — 经营经验沉淀机制

> 定义从运营数据到长期知识的完整沉淀工作流。
>
> 核心目标：**系统必须持续学习，避免聊天记录堆积式记忆。**

---

## 总体流程

```
┌─────────────────────────────────────────────────────────────┐
│                    经验沉淀管道                                │
├──────────┬──────────┬──────────┬──────────┬─────────────────┤
│ 运营数据  │ → 发现   │ → 分析   │ → 验证   │ → 长期知识       │
│ (原料)   │ (信号)   │ (加工)   │ (质检)   │ (成品)          │
├──────────┼──────────┼──────────┼──────────┼─────────────────┤
│ 02_Ops   │ 规律/异常 │ 分析提炼 │ 验证确认 │ 01_Knowledge    │
│          │          │          │          │ 00_Strategy     │
└──────────┴──────────┴──────────┴──────────┴─────────────────┘
```

## 管道 1：运营数据 → 最佳实践

### 触发条件
- 某操作连续 5 次以上成功 → 初步判断为有效模式
- 运营效率显著提升 → 记录并分析原因
- 产量/质量持续稳定 → 归纳操作条件

### 执行流程

```
Step 1: 识别模式
  从 02_Operations 中发现重复出现的成功操作序列

Step 2: 形成草案
  将模式整理为初步操作指南，标注"待验证"

Step 3: 验证
  至少在 3 个不同条件下重复验证该模式

Step 4: 沉淀
  验证通过 → 写入 01_Knowledge 作为最佳实践
  验证失败 → 记录失败原因，标注"无效模式"
```

### 输出示例
```
01_Knowledge/Best_Practice_育肥期温度控制_v1.md
```

## 管道 2：运营异常 → Incident → 经验

### 触发条件
- 设备故障、疫病、供应链中断
- 连续 2 次以上出现同一类操作错误
- 数据指标超出正常范围

### 执行流程

```
Step 1: 事件记录
  → 写入 03_Incidents/YYYY-MM-DD_事件类型_简述.md
  → 使用 Incident_Template

Step 2: 响应处理
  → 执行紧急应对措施
  → 记录采取措施和效果

Step 3: 根因分析
  → 使用 5 Whys 或鱼骨图
  → 确认根本原因

Step 4: 方案制定与执行
  → 制定解决方案
  → 执行并验证效果

Step 5: 复盘
  → 回答复盘 6 问（见 Memory_Rules 6.2）
  → 评估是否需要更新 SOP

Step 6: 经验沉淀（强制）
  → 将经验总结写入 01_Knowledge
  → 如果需要，更新相关 SOP（版本 +1）
  → 标记 03_Incidents 事件为"已关闭-经验已沉淀"
```

### 流程图

```
02_Operations 异常信号
    ↓
03_Incident 创建 ──────────────────┐
    ↓                               │
应急处理                             │
    ↓                               │
根因分析                             │  ← 复盘
    ↓                               │
解决方案                             │
    ↓                               │
效果验证                             │
    ↓                               │
复盘 ───→ 01_Knowledge 经验沉淀 ────┘
                ↓
          SOP 更新 (如果需要)
```

## 管道 3：日常决策 → 决策依据 → 战略

### 触发条件
- 重大采购/销售决策
- 投资决策
- 经营方向调整
- 风险应对策略选择

### 执行流程

```
Step 1: 决策记录
  → 记录决策内容、依据、预期结果

Step 2: 执行追踪
  → 记录执行过程和实际结果

Step 3: 效果评估
  → 对比预期 vs 实际
  → 分析偏差原因

Step 4: 战略沉淀
  → 验证有效的决策模式 → 写入 00_Strategy
  → 失败决策 → 写入 03_Incidents → 走管道 2
```

---

## 管道 4：Agent 运行 → 系统优化

### 触发条件
- Agent 执行失败率超过阈值
- Agent 决策质量评估不达标
- 新能力上线后评估

### 执行流程

```
Step 1: 收集失败/低效案例
  → 从 04_Agent_Runtime 提取

Step 2: 分类分析
  → 工具问题？记忆问题？推理问题？

Step 3: 优化
  → 更新 Agent 配置、提示词或流程

Step 4: 验证
  → 跟踪优化后效果

Step 5: 记录
  → 优化经验写入 01_Knowledge Agent 相关章节
```

## 沉淀质量控制

| 检查项 | 标准 | 截停条件 |
|--------|------|----------|
| 可复现性 | 至少验证 3 次 | 无法复现 → 标注"待验证" |
| 因果关系 | 明确原因 → 结果 | 仅发现相关性 → 标注"需进一步分析" |
| 可执行性 | 步骤清晰，可操作 | 步骤模糊 → 补充细节后再沉淀 |
| 时效性 | 当前仍然有效 | 已过时 → 标注"已过期" |
| 唯一性 | 不与已有知识矛盾 | 矛盾 → 验证哪个更准确，更新知识 |

---

## 避免常见的沉淀陷阱

| 陷阱 | 现象 | 对策 |
|------|------|------|
| 堆积式记忆 | 所有聊天记录都存成"记忆" | 只有经过管道处理的才进入长期记忆 |
| 未经验证的经验 | 凭感觉总结 | 标注"待验证"，验证通过后正式沉淀 |
| 过时知识 | 旧 SOP 和新做法矛盾 | 版本管理 + 定期审核 |
| 重复沉淀 | 同一经验记多次 | 写入前搜索已有知识库 |
| 过度抽象 | 经验太宽泛无法执行 | 保持具体，附案例 |

## 定期总结机制

| 周期 | 输入 | 输出 | 目标目录 |
|------|------|------|----------|
| 月度 | 02_Operations 数据 | 月度运营总结 | 01_Knowledge |
| 季度 | 03_Incidents 复盘 | 季度风险总结 | 01_Knowledge |
| 半年度 | 所有目录 | 半年度经营回顾 | 00_Strategy |
| 年度 | 全库 | 年度复盘与规划 | 00_Strategy |

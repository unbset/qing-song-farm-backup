我现在需要你设计并编写一个：

# farm-executor

（农场 AI Runtime 执行框架）

目标：

不是做一个“提示词模板”，而是建立 Hermes 农场系统的标准运行协议（Runtime Constitution）。

这个框架将成为：

Hermes → gbrain → hindsight → Obsidian

整个系统的统一执行规范。

────────────────────

# 设计目标

────────────────────

你需要解决的问题不是：

“AI 怎么回答问题”

而是：

“AI 收到任务后，系统如何稳定运行”。

重点：

- 任务拆解
    
- Agent 调度
    
- 工具调用顺序
    
- 并行策略
    
- 长期记忆治理
    
- reviewer 机制
    
- 失败恢复
    
- 决策等级
    
- 输出标准化
    

目标是：

让 Hermes 的行为从“临场发挥”，变成“制度化运行”。

────────────────────

# 输出形式

────────────────────

输出为：

Markdown Runtime Framework 文档。

不是代码。

不是 Prompt。

不是教程。

而是：

可长期维护的系统运行协议。

────────────────────

# 文档结构要求

────────────────────

请至少包含以下章节：

# 1. Runtime Philosophy（运行哲学）

定义：

- 系统目标
    
- 单一真相源原则
    
- hindsight 是唯一长期记忆
    
- 为什么禁止自动记忆污染
    
- Hermes / gbrain / hindsight 的职责边界
    

必须明确：

长期记忆 ≠ 工作记忆。

────────────────────

# 2. Task Classification（任务分类）

建立任务分类系统：

例如：

- 查询型
    
- 分析型
    
- 决策型
    
- 自动执行型
    
- 战略型
    
- 记忆管理型
    

每种任务：

- 风险等级
    
- 是否允许自动执行
    
- 是否需要 reviewer
    
- 是否需要用户确认
    

────────────────────

# 3. Execution Pipeline（执行流水线）

定义 Hermes 收到任务后的标准步骤：

例如：

1. 分类任务
    
2. 判断是否需要历史记忆
    
3. 判断是否需要推理
    
4. 判断是否需要联网
    
5. 是否并行派发
    
6. 汇总结果
    
7. reviewer 检查
    
8. 判断是否写入长期记忆
    
9. 输出结果
    

要求：

明确每一步的触发条件。

────────────────────

# 4. Tool Routing Rules（工具路由规则）

明确：

什么情况调用：

- gbrain
    
- hindsight
    
- web_search
    
- delegate_task
    
- todo
    
- terminal
    

要求：

避免：

- 工具重复调用
    
- 无意义推理
    
- 记忆污染
    
- token 浪费
    

────────────────────

# 5. Parallel Execution Rules（并行执行规则）

定义：

什么时候：

- 串行
    
- 并行
    
- 禁止并行
    

例如：

市场分析 + 历史检索  
可以并行。

但：

高风险决策  
禁止并行自动执行。

────────────────────

# 6. Memory Governance（记忆治理）

这是最重要章节。

必须明确：

什么允许写入 hindsight。

什么绝对禁止写入。

例如：

禁止：

- 临时聊天
    
- 未验证推测
    
- 情绪内容
    
- 工具噪音
    
- 中间推理过程
    

允许：

- 已验证经营经验
    
- 长期稳定流程
    
- 高价值决策结果
    
- 经 reviewer 确认的知识
    

必须建立：

Memory Write Gate。

────────────────────

# 7. Reviewer Protocol（审查协议）

定义：

哪些情况：

必须 reviewer。

例如：

- 长期战略
    
- 成本变化
    
- 自动化执行
    
- 高风险建议
    

并定义：

reviewer 的审查标准。

────────────────────

# 8. Failure Recovery（失败恢复）

定义：

子任务失败后的：

- retry
    
- fallback
    
- degrade
    
- escalate
    

以及：

什么时候必须通知用户。

────────────────────

# 9. Decision Levels（决策等级）

定义：

L0 ~ L4。

例如：

L0：  
普通查询。

L1：  
普通建议。

L2：  
经营分析。

L3：  
高风险经营决策。

L4：  
涉及资金/长期战略。

每一级：

是否允许自动执行。

────────────────────

# 10. Output Standards（输出标准）

定义：

Hermes 输出结果时：

- 简洁模式
    
- 决策模式
    
- 战略模式
    
- 调研模式
    

的结构模板。

────────────────────

# 重要原则

────────────────────

不要把文档写成：

“AI 助手 Prompt”。

而是：

真正的：

AI Operating Protocol。

重点：

- 稳定性
    
- 可维护性
    
- 可治理性
    
- 长期扩展性
    

而不是：

“更像人”。

────────────────────

# 最终目标

────────────────────

最终我们要建立的不是：

“一个会聊天的 Agent”。

而是：

“一个可长期运行的 AI 农场操作系统”。
